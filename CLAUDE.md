# Claude Project Instructions
このプロジェクトでは、以下の運用ルールに従って作業してください。
## 1. Plan First
- 非自明なタスクは、必ず最初に計画を立ててから進めること
- 目安として、3ステップ以上必要な作業、または設計判断を含む作業は計画対象とする
- 実装だけでなく、調査、比較、検証についても計画を立てること
- 想定外の問題や前提のズレが見つかった場合は、そのまま進めず、いったん止まって再計画すること
- 曖昧なまま進めないよう、最初に仕様・前提・進め方・検証方法を明確化すること
- 要件に複数の解釈がある場合は、サイレントに1つを選ばず、解釈を明示してユーザーに選ばせること
- より簡単・シンプルなアプローチがあると判断したら、実装前に提案・確認すること
## 2. Task Management
- 作業開始時に tasks/todo.md を作成または更新し、チェック可能なTODO形式で計画を書くこと
- 大きい作業は、実行可能な粒度まで分解すること
- 実装前に、何をどの順番で進め、どう確認するかを明確にすること
- 進捗に応じて tasks/todo.md の項目を随時更新し、完了した項目にはチェックを付けること
- 最後に tasks/todo.md の末尾へレビューを追加し、実施内容・確認内容・残課題を記録すること
## 3. Focused Execution
- 複雑な問題は論点ごとに分割し、1つの作業単位には1つの目的だけを持たせること
- 調査、探索、比較、検証は可能な限り分離して扱うこと
- メインの流れでは、常に全体像が見える状態を保つこと
- 不要に文脈を混ぜず、シンプルで追いやすい進行を維持すること
## 4. Verification Before Done
- 完了と判断する前に、必ず妥当な検証を行うこと
- 必要に応じて、変更前後の挙動差分を確認すること
- テスト、ログ確認、表示確認、出力確認など、適切な方法で正しさを示すこと
- 常に「この内容でスタッフエンジニアが見ても承認できるか」を基準に確認すること
- 検証できていない場合は、完了扱いにしないこと
## 5. Demand Elegance
- 非自明な変更では、一度「もっとシンプルで保守しやすい方法はないか」を検討すること
- 修正が場当たり的・継ぎ足し的になりそうなら、より自然で一貫性のある解決策を優先すること
- ただし、単純な変更に対して過剰設計はしないこと
- シンプルさ、可読性、保守性のバランスを取ること
## 6. Autonomous Bug Fixing
- バグ報告を受けたら、細かい指示待ちをせず、自律的に原因特定から修正・検証まで進めること
- エラー、ログ、失敗テストなど根拠を押さえて対応すること
- 一時しのぎではなく、根本原因の解決を優先すること
- 必要に応じて、関連する不具合の再発防止も考慮すること
## 7. Lessons and Self-Improvement
- ユーザーから修正や指摘を受けた場合は、tasks/lessons.md を更新すること
- tasks/lessons.md には、ミスの内容、原因、再発防止ルールを記録すること
- 以後の作業では、その教訓を反映し、同じ失敗を繰り返さないこと
- セッション開始時または関連作業の開始時には、必要に応じて tasks/lessons.md を確認すること
## 8. Change Communication
- 各ステップで、以下を簡潔に共有すること
  - 何を変えたか
  - なぜ変えたか
  - 何を確認したか
- 説明は高い粒度で簡潔にまとめること
- 不要に細かい実況は避けるが、重要な判断は省略しないこと
## Core Principles
- 常にシンプルさを優先する
- 変更範囲は最小限に抑える
- 必要な箇所だけを変更する
- 手を抜かず、根本原因を特定して対応する
- 応急処置より、保守しやすい解決を選ぶ
- 新しいバグを生まないことを重視する
- 実装・設計・検証の妥当性を、自分で疑って確認する
- 依頼と直接関係のない既存コード・コメント・書式は変更しない（Surgical Changes）
- 自分の変更が生んだ不要コードのみ削除し、既存のデッドコードは削除せず指摘にとどめる
- 変更した全行が、ユーザーの依頼に直接トレースできることを確認する

## デザインルール
- UIを実装する際は、必ず `DESIGN.md` を参照すること
- `DESIGN.md` に定義されていない色・フォント・余白を独自に追加しないこと
- ボタンのスタイルは `Component Stylings` の定義に従うこと
- カード、入力欄、CTAブロック、ヘッダー、フッターも `DESIGN.md` の定義に従うこと
- 新しいUIコンポーネントを追加する場合は、先に `DESIGN.md` にルールを追加してから実装すること
- LINE登録CTAを最重要導線として扱うこと
- Shopifyのカート・決済・商品管理・ログイン機能は実装しないこと
